## Решение #1: Выбор фреймворка FastAPI

**Проблема:** Нужен лёгкий, современный веб-фреймворк для быстрого создания API и HTML-страниц.

**Рассматривали:**
- Flask — простой, но требует дополнительных расширений
- Django — мощный, но избыточен для небольшого проекта
- FastAPI — современный, асинхронный, с автоматической документацией

**Решение:** ✅ **FastAPI**

**Обоснование:**
- Встроенная поддержка асинхронности
- Автоматическая генерация OpenAPI-документации
- Валидация данных через Pydantic
- Отличная производительность
- Простая интеграция с Jinja2 для рендеринга HTML

**Последствия:**
- Меньше кода для валидации запросов
- Встроенная документация API по адресу `/docs`
- Легко расширять функционал

## Решение #2: База данных SQLite вместо PostgreSQL

**Проблема:** Выбор СУБД для хранения данных.

**Рассматривали:**
- PostgreSQL — производственная СУБД, требует настройки
- MySQL — аналогично PostgreSQL
- SQLite — файловая БД, не требует сервера

**Решение:** ✅ **SQLite**

**Обоснование:**
- Проект запускается локально (требование ТЗ: "Покупать VPS/сервер не нужно")
- Не требует установки дополнительного ПО
- Достаточно для демонстрации функционала
- Упрощает развёртывание для проверяющего

**Последствия:**
- Файл БД хранится локально (`repair_db.sqlite3`)
- Не подходит для высокой нагрузки (но это не требуется для тестового задания)
- Добавлен в `.gitignore` для исключения из репозитория

## Решение #3: Оптимистичная блокировка для защиты от Race Condition

**Проблема:** Два мастера могут одновременно нажать "Взять в работу" на одну заявку.

**Рассматривали:**
- Пессимистичная блокировка (`SELECT ... FOR UPDATE`) — сложно, требует транзакций
- Очередь запросов — усложняет архитектуру
- Оптимистичная блокировка через поле `version` — просто и эффективно

**Решение:** ✅ **Оптимистичная блокировка через поле `version`**

**Обоснование:**
- Не требует блокировок на уровне БД
- Простая реализация: увеличиваем `version` при каждом изменении
- Позволяет детектировать конфликты на уровне приложения
- Возвращаем `409 Conflict` при попытке изменить устаревшую версию

**Реализация:**
```python
# Проверка статуса перед изменением
if req.status != RequestStatus.assigned:
    raise HTTPException(status_code=409, detail="Заявка уже взята в работу")

# Увеличение версии
req.version += 1
req.status = RequestStatus.in_progress
db.commit()

Последствия:
Один запрос успешен (303 Redirect), второй получает 409 Conflict
Защита проверена автотестом test_race_condition_take_request
Скрипт race_test.ps1 для ручной проверки
Решение #4: Упрощённая авторизация через URL-параметры
Проблема: Нужен способ идентификации пользователя без сложной системы аутентификации.
Рассматривали:
JWT-токены — избыточно для тестового задания
Сессии через cookies — требует настройки SessionMiddleware
URL-параметр user_id — просто и прозрачно
Решение: ✅ URL-параметр user_id
Обоснование:
Не требует реализации системы логина/пароля
Прозрачно для проверяющего (видно в URL)
Упрощает тестирование (можно быстро переключаться между пользователями)
Достаточно для демонстрации ролевой модели
Реализация:
# Извлечение user_id из URL
def get_user_id(request: Request) -> int | None:
    uid = request.query_params.get("user_id")
    return int(uid) if uid and uid.isdigit() else None

Последствия:
Не подходит для продакшена (нет защиты от подделки)
Для тестового задания — приемлемо
Упрощает навигацию между ролями (диспетчер/мастер)
Решение #5: Монолитная структура проекта
Проблема: Как организовать структуру файлов для небольшого проекта.
Рассматривали:
Микросервисы — избыточно для 3 страниц
Модульная архитектура с сервисами — усложняет
Монолит с разделением на роуты — оптимально
Решение: ✅ Монолит с разделением на роуты
Структура:
app/
├── main.py           # Точка входа, настройка FastAPI
├── models.py         # SQLAlchemy модели
├── database.py       # Настройки подключения к БД
├── routers/
│   └── requests.py   # Все роуты для заявок
└── templates/
    └── base.html     # Единый HTML-шаблон

Обоснование:
Все роуты в одном файле (requests.py) — легко найти и изменить
Единый шаблон base.html — согласованный UI
Минимум файлов — быстро разобраться
Последствия:
Легко масштабировать: можно вынести роуты в отдельные файлы
Подходит для проекта до 10-15 страниц
Решение #6: SQLAlchemy ORM вместо сырых SQL-запросов
Проблема: Как работать с базой данных.
Рассматривали:
Сырые SQL-запросы — полный контроль, но много кода
SQLAlchemy Core — гибко, но сложнее
SQLAlchemy ORM — удобно, типизировано
Решение: ✅ SQLAlchemy ORM
Обоснование:
Типизированные модели данных
Автоматическое создание таблиц через Base.metadata.create_all()
Удобные запросы через .query()
Интеграция с Pydantic для валидации
Реализация:
class ServiceRequest(Base):
    __tablename__ = "service_requests"
    
    id = Column(Integer, primary_key=True, index=True)
    client_name = Column(String, nullable=False)
    status = Column(Enum(RequestStatus), default=RequestStatus.new)
    assigned_to = Column(Integer, ForeignKey("users.id"), nullable=True)
    version = Column(Integer, default=1)

Последствия:
Меньше кода для CRUD-операций
Автоматическая миграция схем (при изменении моделей)
Легко добавлять новые поля
Решение #7: Автотесты через pytest + TestClient
Проблема: Как проверить функционал автоматически.
Рассматривали:
unittest — встроен в Python, но многословный
pytest — лаконичный, с фикстурами
Ручное тестирование — долго, не воспроизводимо
Решение: ✅ pytest + FastAPI TestClient
Обоснование:
Лаконичный синтаксис тестов
Фикстуры для setup/teardown БД
TestClient эмулирует HTTP-запросы без запуска сервера
6 тестов покрывают весь критический функционал
